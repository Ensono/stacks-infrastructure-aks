== Troubleshooting

=== PAT Validation Failures

Symptoms:

* `setup:validate:azdo:pat` fails
* Variable group creation fails

Checks:

* Verify scopes: `vso.project`, `vso.variablegroups_manage`
* Confirm PAT is not expired

=== Terraform Init/Plan Issues

Symptoms:

* Backend init fails
* Plan cannot be created

Checks:

* Validate `TF_BACKEND_INIT` string
* Confirm storage account and container permissions
* Ensure workspace name matches `TF_VAR_stage`

=== Variable Group Creation Errors

Symptoms:

* `azuredevops` provider fails

Checks:

* PAT scope includes Variable Groups Manage
* `TF_VAR_create_ado_variable_group=true`

=== DNS Propagation Problems

Symptoms:

* DNS records not updated

Checks:

* Validate `TF_VAR_dns_resource_group` and zone names
* Confirm NS delegation if using parent zones

=== InSpec Test Failures

Symptoms:

* `tests:infra:run` fails

Checks:

* Ensure `tests:infra:inputs` runs before `tests:infra:run`
* Validate `/eirctl/inspec_inputs.yml` contents
* Confirm Azure credentials are present

=== Terraform Workspace Selection Fails

Symptoms:

* `tests:infra:inputs` fails with "No workspace name specified to create or switch to"

Checks:

* Ensure `TF_VAR_stage` environment variable is set (e.g., `dev`, `nonprod`, `prod`)
* Run `source ./.eirctl/envvar-azure-dev.sh` to load required environment variables
* Verify `TF_FILE_LOCATION` points to `/eirctl/deploy/terraform` (defaults automatically if not set)
* Run `eirctl run setup:environment` to validate all required variables are present

=== Helm Deployment Errors

Symptoms:

* `infra:helm:apply` fails

Checks:

* Validate TFOUT_ variables are present
* Confirm chart values template paths
* Verify cluster credentials and network access

=== Default Node Pool Update Failures

Symptoms:

* Terraform plan/apply fails with: `temporary_name_for_rotation` must be specified when updating default_node_pool properties

Cause:

Azure provider requires `temporary_name_for_rotation` when modifying certain default node pool properties (vm_size, os_disk_size_gb, enable_auto_scaling, etc.). This enables zero-downtime node rotation.

Resolution:

The AzureRM provider requires `temporary_name_for_rotation` to be set when rotating the default node pool. The Ensono Stacks AKS module (v8.0.29 and later, as used in this repository) exposes this via the `temporary_name_for_rotation` Terraform variable. To resolve:

1. **For autoscaling changes only**: The infrastructure as code is configured to set `min_nodes` and `max_nodes` to `null` when autoscaling is disabled, which complies with AzureRM provider v4.x requirements. If the error persists:
   - Set the `temporary_name_for_rotation` variable (for example via `TF_VAR_temporary_name_for_rotation`) to a short, unique temporary node pool name
   - Run the standard `eirctl`/`Invoke-Terraform` workflow (do not call `terraform` directly) to apply the change
   - Once the rotation has completed successfully, unset or remove `temporary_name_for_rotation` for subsequent runs
   - As a fallback, you can taint the node pool to force replacement using the standard `eirctl`/`Invoke-Terraform` workflow:
     `Invoke-Terraform -TaintTarget 'module.aks_bootstrap.azurerm_kubernetes_cluster.default[0]'` (see `build/eirctl/tasks.yaml` for task definitions)

2. **For vm_size or disk changes**:
   - Set the `temporary_name_for_rotation` Terraform variable (for example via `TF_VAR_temporary_name_for_rotation`) before running plan/apply so that the provider can create a temporary node pool and perform zero-downtime rotation
   - After a successful apply and verification, unset or remove `temporary_name_for_rotation` to avoid unintended future rotations
   - If automation cannot be used, coordinate a manual node pool update in the Azure portal (for example, by creating a temporary node pool, draining workloads, and deleting the old pool once workloads are migrated)
